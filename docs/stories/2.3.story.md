# Story 2.3: Manifest Cache

## Status

Draft

## Story

**As a** developer,
**I want** cached OCI manifests readable from the store,
**so that** the resolver can work without network access.

## Acceptance Criteria

1. `cache_manifest()` stores in `manifests/{registry}/{repo}/{tag}/{platform}.json` with `_buncker.cached_at`
2. `get_manifest(registry, repository, reference, platform)` returns cached manifest or `None`
3. Digest reference lookup via `_buncker.source_digest`
4. Tests: cache, retrieve, miss -> None, digest lookup

## Tasks / Subtasks

- [ ] Task 1: Implement `cache_manifest()` (AC: 1)
  - [ ] Accept: registry, repository, tag, platform, manifest data (dict/JSON)
  - [ ] Write to `manifests/{registry}/{repository}/{tag}/{platform}.json`
  - [ ] Add `_buncker` extension object with `cached_at` (ISO timestamp) and `source_digest` (sha256 of original manifest)
  - [ ] Create directory structure as needed
  - [ ] Atomic write (temp + rename)
- [ ] Task 2: Implement `get_manifest(registry, repository, reference, platform)` (AC: 2)
  - [ ] Accept: registry, repository, reference (tag or digest), platform (e.g. `linux/amd64`)
  - [ ] If reference is a tag: look up `manifests/{registry}/{repository}/{tag}/{platform}.json`
  - [ ] If reference is a digest: scan cached manifests for matching `_buncker.source_digest`
  - [ ] Return parsed manifest dict or `None` if not found
- [ ] Task 3: Implement digest reference lookup (AC: 3)
  - [ ] When looking up by digest, iterate manifest files and check `_buncker.source_digest`
  - [ ] Return first match or `None`
- [ ] Task 4: Write tests (AC: 4)
  - [ ] Test cache_manifest writes to correct path with correct structure
  - [ ] Test get_manifest retrieves cached manifest by tag
  - [ ] Test get_manifest returns None for miss
  - [ ] Test digest lookup finds cached manifest by source_digest
  - [ ] Test _buncker extension is present with cached_at and source_digest

## Dev Notes

### Manifest Cache Structure

Cached manifests are stored under the store path:

```
manifests/{registry}/{repository}/{tag}/{platform}.json
```

Example: `manifests/docker.io/library/nginx/1.25/linux-amd64.json`

**Platform convention:** The `platform` parameter uses OCI format (`linux/amd64`) but is converted to filesystem-safe format by replacing `/` with `-` (e.g. `linux/amd64` -> `linux-amd64`). This conversion is internal to `cache_manifest()` and `get_manifest()`.

Each cached manifest has a `_buncker` extension object added:
```json
{
  "schemaVersion": 2,
  "mediaType": "...",
  "config": {...},
  "layers": [...],
  "_buncker": {
    "cached_at": "2026-03-04T10:00:00Z",
    "source_digest": "sha256:abc123..."
  }
}
```

`_buncker` is the only allowed OCI extension key in the project.

[Source: `docs/architecture/9-storage-schema.md`, `docs/architecture/13-coding-standards.md`]

### Component Context

This is the offline `registry_client` - it reads cached manifests only. NO network requests allowed in this module. The manifests are initially cached during response import (Story 3.3).

[Source: `docs/architecture/5-components.md#buncker/registry_client`]
