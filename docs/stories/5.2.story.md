# Story 5.2: End-to-End Tests

## Status

Draft

## Story

**As a** developer,
**I want** automated e2e tests covering the full USB cycle,
**so that** we can release with confidence.

## Acceptance Criteria

1. E2E: setup -> pair -> analyze -> generate -> fetch (mock) -> import -> pull -> blob verified
2. E2E: key rotation -> new request -> fetch with new key -> import OK
3. E2E: corrupt blob in response -> partial import, valid blobs kept
4. E2E: invalid HMAC -> total refusal
5. All tests run in CI without real network (mock HTTP servers)
6. Total coverage >= 80%

## Dev Notes

### E2E Test Architecture

E2E tests simulate the full USB cycle without real network. All external registries are replaced by a local mock HTTP server.

```
Mock Registry (localhost)
    ^
    |  (HTTP)
buncker-fetch (online side)
    |
    v (encrypted files)
buncker (offline side)
    |
    v (OCI API)
Docker pull simulation
```

[Source: `docs/architecture/14-test-strategy.md`]

### Test Pyramid

- 70% unit, 25% integration, 5% e2e
- E2E tests are the 5% - they cover critical paths, not every edge case
- Edge cases should already be covered by unit and integration tests

[Source: `docs/architecture/14-test-strategy.md`]

### CI Pipeline

Full pipeline: `ruff check` -> `ruff format --check` -> `pytest (unit)` -> `pytest (integration)` -> `pytest (e2e)` -> `coverage`

E2E tests must run without real network. Use mock HTTP servers on localhost.

[Source: `docs/architecture/14-test-strategy.md`]

### Core Workflows Tested

1. **Workflow 1:** Dockerfile analysis + request generation [Source: `docs/architecture/7-core-workflows.md`]

3. **Workflow 3:** Import response [Source: `docs/architecture/7-core-workflows.md`]
