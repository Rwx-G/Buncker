# Story 1.4: Exception Hierarchy & Logging Setup

## Status

Ready for Review

## Story

**As a** developer,
**I want** a consistent error and logging framework from day one,
**so that** all subsequent modules use the same patterns.

## Acceptance Criteria

1. `shared/exceptions.py` defines: `BunckerError`, `ConfigError`, `CryptoError`, `StoreError`, `ResolverError`, `RegistryError`, `TransferError`
2. Each exception accepts a message string and an optional context dict
3. `shared/logging.py` configures a JSON Lines logger: each entry contains `ts`, `event`, `level`, and additional fields
4. Helper `setup_logging(level, output_path)` configures the handler (JSONL file + stderr for ERROR+)
5. Tests: exceptions raise correctly, logger produces valid JSON

## Dev Notes

### Exception Hierarchy

```python
class BunckerError(Exception):
    def __init__(self, message: str, context: dict | None = None): ...

class ConfigError(BunckerError): ...
class CryptoError(BunckerError): ...
class StoreError(BunckerError): ...
class ResolverError(BunckerError): ...
class RegistryError(BunckerError): ...
class TransferError(BunckerError): ...
```

All errors must be **actionable**: what failed + context + what to do.

[Source: `docs/architecture/12-error-handling-strategy.md`]

### Logging Format

**JSON Lines format**, each line is valid JSON:
```json
{"ts": "2026-03-04T10:30:00Z", "event": "blob_imported", "level": "INFO", "digest": "sha256:abc..."}
```

**Standard events:** `dockerfile_analyzed`, `transfer_manifest_generated`, `transfer_imported`, `blob_pulled`, `blob_missing`, `gc_candidate`, `gc_executed`, `key_rotation`

**Levels:** DEBUG, INFO, WARNING, ERROR

**Security:** Never log mnemonic, derived keys, Bearer tokens, passwords.

[Source: `docs/architecture/12-error-handling-strategy.md#Logging`]

### Log File Locations

- Offline: `/var/lib/buncker/logs/buncker.jsonl`
- Online: `~/.buncker/logs/fetch.jsonl`

These paths are configured at runtime. The logging module just needs to accept `output_path` as parameter.

[Source: `docs/architecture/9-storage-schema.md`]

### Migration from Story 1.3

> **Action required:** Story 1.3 (OCI Module) uses stdlib exceptions (`ValueError`, `json.JSONDecodeError`, `FileNotFoundError`) as a temporary measure because 1.4 does not exist yet at that point. Once this story is implemented, review `shared/oci.py` and migrate error cases to the appropriate `BunckerError` subclasses where it makes sense (e.g. invalid manifest -> `RegistryError`, invalid blob digest -> `StoreError`). Keep stdlib exceptions for truly stdlib-level errors (malformed JSON, missing file) and use `BunckerError` subclasses for domain-level errors.

### Coding Standards

- Use stdlib `logging` module
- `pathlib.Path` for the output path
- Type hints on public signatures
- The `JsonFormatter` should use `json.dumps` - no external dependency

[Source: `docs/architecture/13-coding-standards.md`]

### Testing

- **Test locations:** `tests/shared/test_exceptions.py`, `tests/shared/test_logging.py`
- **Framework:** pytest 8.x
- **For logging tests:** Use `tmp_path` fixture for log file, read and parse JSON lines after logging
- **Coverage:** 80% minimum
