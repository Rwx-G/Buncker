# Story 4.4: CLI buncker-fetch

## Status

Done

## Story

**As an** operator,
**I want** a complete CLI to manage the online side,
**so that** I can pair, inspect, fetch, and manage the cache.

## Acceptance Criteria

1. `buncker-fetch pair`: enter 12 words, derive keys, save config
2. `buncker-fetch inspect`: decrypt, display summary
3. `buncker-fetch fetch`: full cycle with `--output` and `--parallelism` options
4. `buncker-fetch status`: cache state
5. `buncker-fetch cache clean`: cleanup
6. Human-readable output by default, `--json` for machine output
7. All errors actionable with context
8. Tests: each subcommand tested

## QA Results

### Review Date: 2026-03-04
### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

`__main__.py` is well-organized with clear separation between subcommand handlers, output formatting, and key derivation. The `_build_parser()` function centralizes all argparse configuration. The `--json` and `--config` flags are global and correctly threaded to all subcommands. `cmd_pair` validates word count and BIP-39 membership before calling `derive_keys`, then stores only a verifiable marker (never the keys or mnemonic). `cmd_fetch` correctly groups blobs by registry before creating per-registry `RegistryClient` instances, and normalizes `docker.io` to `registry-1.docker.io`. `cmd_cache_clean` validates the `--older-than` format with clear error messages. All errors go to stderr; normal output to stdout.

One observation: `__main__.py` imports `_DEFAULT_CONFIG_PATH` from `config.py` (a private symbol). This is acceptable as both files are in the same package, but the leading underscore signals it is not part of the public API.

`config.py` is minimal and correct. `load_config` merges defaults so missing keys are always populated. `validate_config` provides actionable error messages directing the user to run `pair` first.

### Improvements Checklist

- [ ] Minor: `_derive_keys_from_config` prompts for the mnemonic on every `inspect` and `fetch` call. For a batch workflow this is a UX friction point. A session-level key cache (in-memory only) could improve ergonomics in a future story.
- [ ] Minor: `cmd_fetch` hardcodes `_DEFAULT_CACHE_PATH` inside the function body rather than accepting it as a parameter, making tests rely on patching the module-level symbol. A dependency-injection approach (cache path from args or config) would be cleaner.
- [ ] Minor: `_print_output` formats list values as comma-joined strings in human mode. For the `registries` field in `inspect` output this works, but the formatting is generic and may not suit all future fields.
- [ ] Note: `__main__.py` imports `_DEFAULT_CONFIG_PATH` (private symbol) from `config.py`. Acceptable within the same package, but consider exporting it as a public constant if reused elsewhere.

### Security Review

- Mnemonic entered interactively via `input()`, not via CLI argument - avoids shell history exposure. PASS
- Salt generated with `os.urandom(32)` on each `pair` call. PASS
- `derived_key_check` stores an encrypted marker, not the mnemonic or derived keys. PASS
- Mnemonic verification uses `decrypt()` which raises `CryptoError` on wrong key - no timing oracle. PASS
- Credentials for registries are loaded via `load_credentials()` from env vars, never stored in config. PASS
- No secrets appear in log output. PASS

### Performance Considerations

- `derive_keys` runs PBKDF2-SHA256 with 600,000 iterations on each `inspect` and `fetch` invocation (mnemonic re-entry). This is ~1-2 seconds on modern hardware and is intentional for brute-force resistance. Acceptable.
- CLI is I/O bound; no performance concerns in argument parsing or output formatting.
