# Story 4.2: Fetcher & Cache

## Status

Done

## Story

**As an** operator,
**I want** downloaded blobs cached locally and parallel downloads,
**so that** transfers are fast and resumable.

## Acceptance Criteria

1. Cache manages `~/.buncker/cache/blobs/sha256/`
2. `has_blob`/`get_blob`/`store_blob` with SHA256 verification
3. Fetcher downloads blob list with `ThreadPoolExecutor`
4. Cached blobs auto-skipped (resume)
5. SHA256 mismatch -> retry 3x -> errors list
6. Progress reporting
7. `cache clean --older-than Nd`
8. Tests: cache hit/miss, parallelism, corrupt blob retried

## Dev Notes

### Cache Storage

```
~/.buncker/
  cache/blobs/sha256/{digest_hex}
```

Same atomic write pattern as the offline store: temp + verify SHA256 + rename.

[Source: `docs/architecture/9-storage-schema.md`]

### Concurrency

Use `concurrent.futures.ThreadPoolExecutor` - stdlib, simple, sufficient for I/O-bound downloads.

Default parallelism: 4 workers (configurable via `--parallelism` CLI flag).

[Source: `docs/architecture/3-tech-stack.md`]

### Expected Interfaces

The fetcher depends on these interfaces from Story 4.1:

```python
# buncker_fetch/registry_client.py (Story 4.1)
def fetch_blob(self, digest: str) -> Iterator[bytes]:
    """Fetch blob content as an iterator of byte chunks."""
```

The `FetchResult` return type:

```python
@dataclass
class FetchResult:
    downloaded: list[str]   # digests successfully fetched
    skipped: list[str]      # digests already in cache
    errors: list[dict]      # [{"digest": ..., "error": ...}]
```

### Critical Rules

- SHA256 verify on every blob write - no exceptions
- Atomic writes only

[Source: `docs/architecture/13-coding-standards.md`]
