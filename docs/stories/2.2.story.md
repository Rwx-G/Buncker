# Story 2.2: Store GC

## Story

**As an** operator,
**I want** to identify and clean up unused blobs,
**so that** disk space is managed without risking active images.

## Acceptance Criteria

1. `gc_report(inactive_days)` scans meta sidecars and returns candidates list
2. Shared blob is candidate only if its `last_requested` is older than `inactive_days` (blob-level proxy - see Dev Notes)
3. `gc_execute(digests, operator)` deletes blobs and sidecars, logs each deletion
4. `gc_execute` refuses to delete a digest not in the latest report
5. Tests: active blob not candidate, shared blob protected, effective deletion, log produced

## Dev Notes

### GC Design

GC is **manual only** - never automatic deletion. The operator runs a report first, reviews, then explicitly executes. This is a safety feature.

[Source: `docs/architecture/5-components.md#buncker/store` - "GC is admin-triggered, never automatic"]

Flow:
1. `gc_report(inactive_days)` -> list of candidates
2. Operator reviews candidates
3. `gc_execute(digests, operator)` -> deletions

The report is stored in memory (instance attribute) and must match the execute call. If a new report is generated, the previous one is invalidated. This prevents stale reports from being executed after store state has changed.

[Source: `docs/architecture/5-components.md#buncker/store` - report-then-execute pattern is the GC contract]

### Blob Activity Model

The data model (`docs/architecture/4-data-models.md`) defines a single `last_requested` timestamp per blob (not per `image_ref`). This means GC activity is determined at the blob level: if any client has pulled the blob recently (updating `last_requested`), the blob is considered active regardless of how many images reference it. The `image_refs` list is informational for the GC report but does not drive the activity decision.

### Logging Events

- Event `gc_candidate`: logged during report generation
- Event `gc_executed`: logged during deletion with `digest`, `size`, `operator`
- Use the JSON Lines logger from `shared/logging.py` (Story 1.4)

[Source: `docs/architecture/12-error-handling-strategy.md#Logging`]
