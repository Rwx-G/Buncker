## Dev Notes

### OCI Specifications

The OCI module lives in `shared/oci.py` and is used by both packages. It implements the OCI Image Spec (not Distribution Spec - that's the HTTP API).

**OCI Image Layout structure:**
```
oci-layout          # {"imageLayoutVersion": "1.0.0"}
index.json          # Root index referencing manifests
blobs/sha256/       # Content-addressable blob store
```

**OCI Manifest v2 structure (application/vnd.oci.image.manifest.v1+json):**
```json
{
  "schemaVersion": 2,
  "mediaType": "application/vnd.oci.image.manifest.v1+json",
  "config": {"mediaType": "...", "digest": "sha256:...", "size": 1234},
  "layers": [{"mediaType": "...", "digest": "sha256:...", "size": 5678}]
}
```

**OCI Image Index structure (application/vnd.oci.image.index.v1+json):**
```json
{
  "schemaVersion": 2,
  "mediaType": "application/vnd.oci.image.index.v1+json",
  "manifests": [
    {"mediaType": "...", "digest": "sha256:...", "size": 1234, "platform": {"architecture": "amd64", "os": "linux"}}
  ]
}
```

[Source: `docs/architecture/5-components.md#shared/oci`]

### Data Models

The OCI blob in the store has these attributes: `digest`, `size`, `media_type`, `image_refs`, `first_imported`, `last_requested`, `request_count`, `gc_status`. However, the OCI module itself only deals with the spec-level structures (manifest, index, layers). Store-specific metadata is handled by the store module (Epic 2).

[Source: `docs/architecture/4-data-models.md#OCI Blob`]

### Coding Standards

- Use `@dataclass` for all data structures crossing module boundaries
- Type hints on all public signatures
- `pathlib.Path` for file paths
- f-strings only
- SHA256 verify on every blob read/write - no exceptions
- `_buncker` is the only allowed OCI extension key

[Source: `docs/architecture/13-coding-standards.md`]
