# Story 2.5: Resolver Pipeline

## Status

Done

## Story

**As an** operator,
**I want** a complete analysis that tells me exactly which blobs are missing,
**so that** I can decide to generate a transfer request.

## Acceptance Criteria

1. `resolve_dockerfile(path, build_args, store, registry_client)` orchestrates: parse -> manifest lookup -> `list_missing`
2. Returns `AnalysisResult` with resolved images, present/missing blobs, total missing size
3. Deduplication: shared blob counted once
4. Private registries -> skipped with warning
5. Tag `latest` -> warning logged
6. Manifest not in cache -> image marked "unresolved" with explicit message
7. Tests: multi-image with shared blobs, private image skipped, missing manifest signaled

## Dev Notes

### Resolver Pipeline Workflow

```
parse_dockerfile(path, build_args)
    |
    v
For each external image:
    get_manifest(registry, repo, tag) -- from cache
        |
        v
    Extract layer digests from manifest
        |
        v
    store.list_missing(layer_digests)
        |
        v
    Collect missing blobs (deduplicated)
```

[Source: `docs/architecture/7-core-workflows.md#Workflow 1`]

### AnalysisResult Data Model

```python
@dataclass
class AnalysisResult:
    source_path: str
    build_args: dict[str, str]
    images: list[ResolvedImage]
    present_blobs: set[str]       # digests already in store
    missing_blobs: list[dict]     # [{registry, repository, digest, size, media_type}]
    total_missing_size: int       # bytes
    warnings: list[str]
```

No specific architecture doc defines this dataclass - it's derived from the workflow and the PRD acceptance criteria.

## QA Results

### Code Quality Assessment

`resolve_dockerfile` in `buncker/resolver.py` is clean and correctly orchestrates the three-phase pipeline. `AnalysisResult` has all required fields with correct types (`set[str]` for `present_blobs`, `list[dict]` for `missing_blobs`, `list[str]` for `warnings`). Deduplication via `seen_digests: set[str]` correctly spans all images and prevents double-counting. The config layer (manifest `config` key) is included in layer digest extraction via `_extract_layer_digests`, which is architecturally correct - config blobs must also be transferred. Edge cases are all handled: internal images skipped, private images skipped with warning, latest tag warning added without blocking pipeline, missing manifest adds warning and continues. `_find_layer_info` gracefully returns `{}` on miss (size defaults to 0), avoiding a crash when manifest structure is unexpected.

Tests use `FakeCache` and `FakeStore` duck-typed fakes (not mocks) - clean and readable. The `test_deduplication_shared_blobs` test correctly verifies `count == 1` for shared digests. `test_total_missing_size` verifies CONFIG_D=100 + LAYER_A=1000 + LAYER_B=2000 = 3100.

### Performance Considerations

Pipeline is O(images x layers) for digest extraction and O(n) per `list_missing` call. For typical Dockerfiles (2-10 FROM stages, 5-20 layers each) this is negligible. The `seen_digests` set ensures deduplication in O(1) per lookup.
