# Story 1.2: Crypto Module

## Status

Draft

## Story

**As an** operator,
**I want** the cryptographic primitives implemented and tested,
**so that** all transfer security relies on proven, audited code.

## Acceptance Criteria

1. `generate_mnemonic()` returns 12 words from BIP-39 wordlist (2048 words) with `secrets.token_bytes` entropy
2. `derive_keys(mnemonic, salt, iterations=600_000)` returns tuple `(aes_key, hmac_key)` via PBKDF2-SHA256
3. `encrypt(data, aes_key)` encrypts with AES-256-GCM and returns nonce + ciphertext + tag
4. `decrypt(data, aes_key)` decrypts and verifies auth tag. Raises `CryptoError` if invalid
5. `sign(data, hmac_key)` returns HMAC-SHA256 hex digest
6. `verify(data, hmac_key, signature)` returns bool (constant-time comparison)
7. `shared/wordlist.py` contains the complete BIP-39 wordlist (2048 words) embedded
8. Unit tests: round-trip encrypt/decrypt, wrong key -> CryptoError, valid/invalid HMAC, mnemonic has 12 valid words
9. 100% coverage on this module

## Dev Notes

### Crypto Architecture

The crypto module is in `shared/crypto.py` and is used by both `buncker` and `buncker-fetch`. It is the only module that depends on an external package (`python3-cryptography`).

[Source: `docs/architecture/5-components.md#shared/crypto`]

### Implementation Details

**AES-256-GCM:**
- Library: `cryptography.hazmat.primitives.ciphers.aead.AESGCM`
- Nonce: 12 bytes from `os.urandom`
- Wire format: `nonce (12 bytes) || ciphertext || tag (16 bytes)` - GCM appends tag automatically

**PBKDF2:**
- Library: `cryptography.hazmat.primitives.kdf.pbkdf2.PBKDF2HMAC`
- Hash: SHA256
- Iterations: 600,000 (default, configurable)
- Output: 64 bytes total, split into two 32-byte keys (AES key + HMAC key)

**HMAC:**
- stdlib `hmac` + `hashlib.sha256`
- Output: hex digest string
- Comparison: `hmac.compare_digest()` (constant-time)

**BIP-39 Mnemonic:**
- Entropy: `secrets.token_bytes(16)` (128 bits)
- Wordlist: 2048 English words (standard BIP-39)
- Output: 12 space-separated words

[Source: `docs/architecture/15-security.md`, `docs/architecture/5-components.md#shared/crypto`]

### Security Rules

- Never log mnemonic, derived keys, or tokens
- Derived keys are in-memory only during execution, never written to disk
- Config stores only verification hashes and salts
- Mnemonic communicated once via human channel (paper)

[Source: `docs/architecture/15-security.md#Secrets Management`]

### Exception Handling

`CryptoError(BunckerError)` is the exception class for all crypto failures. It accepts a message string and optional context dict. Must be defined in `shared/exceptions.py` (Story 1.4), but can be created inline or imported if 1.4 is done first.

[Source: `docs/architecture/12-error-handling-strategy.md`]

### Testing

- **Test location:** `tests/shared/test_crypto.py`
- **Framework:** pytest 8.x, `unittest.mock` for mocking if needed
- **Coverage requirement:** 100% on this module (critical security code)
- **Test philosophy:** Test-after, but given this is crypto, be thorough
- **Pattern:** Each public function gets its own test class or group
- No mocking of crypto primitives - test real encrypt/decrypt round-trips

[Source: `docs/architecture/14-test-strategy.md`]
