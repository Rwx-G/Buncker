# Story 4.3: Transfer - Decrypt Request & Build Response

## Status

Draft

## Story

**As an** operator,
**I want** buncker-fetch to process requests and build encrypted responses,
**so that** the USB transfer cycle is complete.

## Acceptance Criteria

1. `process_request()` decrypts -> HMAC verify -> parse JSON -> blob list
2. Invalid HMAC -> `CryptoError`, nothing downloaded
3. `build_response()` builds OCI layout -> HMAC -> tar -> encrypt
4. `ERRORS.json` included if blobs failed
5. Default naming: `buncker-response-{datetime}-{source_id}.tar.enc`
6. `buncker_version` check: include newer .deb if available
7. Tests: round-trip with crypto, ERRORS.json case

## Tasks / Subtasks

- [ ] Task 1: Implement `process_request()` (AC: 1, 2)
  - [ ] Read `.json.enc` file
  - [ ] Decrypt with `crypto.decrypt(data, aes_key)`
  - [ ] Extract and verify HMAC signature
  - [ ] If invalid HMAC: raise `CryptoError` immediately, no further processing
  - [ ] Parse JSON to get blob list
  - [ ] Return parsed request: version, source_id, blobs list
- [ ] Task 2: Implement `build_response()` (AC: 3, 4, 5)
  - [ ] Gather blobs from cache
  - [ ] Build OCI Image Layout directory:
    - [ ] `oci-layout` file
    - [ ] `index.json`
    - [ ] `blobs/sha256/{digest_hex}` for each blob
  - [ ] If any blobs failed: create `ERRORS.json` with error details
  - [ ] Create `MANIFEST.sig` (HMAC of blob list)
  - [ ] Tar the directory
  - [ ] Sign with HMAC, then encrypt with AES-256-GCM
  - [ ] Write to `buncker-response-{datetime}-{source_id}.tar.enc`
- [ ] Task 3: Implement version check (AC: 6)
  - [ ] Compare `buncker_version` in request with available .deb
  - [ ] If newer .deb available, include it in the tar
  - [ ] If not available, skip (no error)
- [ ] Task 4: Write tests (AC: 7)
  - [ ] Test process_request decrypts and parses correctly
  - [ ] Test process_request with invalid HMAC raises CryptoError
  - [ ] Test build_response creates valid encrypted tar
  - [ ] Test round-trip: build request (offline) -> process (online) -> build response -> import (offline)
  - [ ] Test ERRORS.json included when blobs fail
  - [ ] Test response naming convention

## Dev Notes

### Request Wire Format

The request file (`.json.enc`) contains:
1. Encrypted payload = `encrypt(json_bytes + "\n" + hmac_hex, aes_key)`

Decryption yields: `json_bytes + "\n" + hmac_hex`
- Split on last newline to separate data and signature
- Verify HMAC of json_bytes with hmac_key
- Parse json_bytes

[Source: `docs/architecture/4-data-models.md#Transfer Request`]

### Response Wire Format

The response file (`.tar.enc`) contains:
1. OCI layout tar + MANIFEST.sig + optional ERRORS.json
2. The entire tar is signed (HMAC), then encrypted

```
tar contents:
  oci-layout
  index.json
  blobs/sha256/{digest}...
  MANIFEST.sig
  ERRORS.json (optional)
```

ERRORS.json format:
```json
[{"digest": "sha256:abc...", "error": "HTTP 404: blob not found on registry"}]
```

[Source: `docs/architecture/4-data-models.md#Transfer Response`]

### Auto-Update Mechanism

The request includes `buncker_version`. If buncker-fetch has access to a newer `.deb`, it can include it in the response tar. The offline operator manually installs it.

[Source: `docs/architecture/11-infrastructure-and-deployment.md#Auto-update`]
